# 🎨 Dify ➕ 本地大模型 ➕ 本地 RAG 闭环极速通关教程

欢迎来到 Dify 本地化大模型应用拼装台！这是一个为本地部署量身定制的避坑实践指南，帮助您用最简单、最稳健的方式打通大模型本地化生态。

---

## 📐 1. 核心架构：“铁三角”各司其职

在整个系统中，各个组件的角色分工极其明确：
1.  **Dify (编排与 UI 交互大厅)**：负责提供高颜值的聊天窗口，并使用工作流画布连接各个服务。
2.  **Llama.cpp 路由器 (大脑)**：跑在宿主机 Windows 上的大模型（如 Qwen/Gemma），提供核心自然语言推理能力。
3.  **我们的 RAG 项目 (外挂硬盘)**：监听本地 8000 端口，专门负责去您的 Obsidian 日志中检索完美的 Markdown 语法片段。

---

## 🎯 任务一：动态设备选择拉起 RAG 硬盘服务

本系统的推理设备（CPU/GPU）已全面支持环境变量动态指定，并提供了命令行交互选择功能。如果您的 GPU 显存已满（例如被本地大模型占满），可以随时灵活切换为 CPU 推理。

1.  **一键启动**：双击运行 [启动RAG服务.bat](file:///E:/project/advanced-rag/启动RAG服务.bat)。
2.  **交互式选择**：
    *   输入 `1` ➡️ 选择 **CPU 模式**（默认。显存占用为 0，防止显卡 OOM 溢出，强烈推荐显卡占满时选择）。
    *   输入 `2` ➡️ 选择 **GPU 模式**（需要 CUDA 支持，速度极快，会占用约 4-6GB 显存）。
3.  **就绪确认**：黑窗口控制台会打印出您选择的运行设备定位，并在本地端口 **`http://127.0.0.1:8000`** 保持监听。

## 🎯 任务二：将本地大模型接入 Dify (终极避坑接入)

由于 Dify 运行在 Docker 容器内部（容器网络隔离），且 C++ 版本的 `llama-server` 没有实现单个模型查询的 API（导致 Dify 的 OpenAI 兼容插件报错 404），我们必须使用以下 **完美避坑方案**：

### 🛠️ 详细接入步骤：
1.  进入 Dify 网页后台，点击右上角头像 ➡️ 选择 **设置 (Settings)** ➡️ 左侧选择 **模型供应商 (Model Provider)**。
2.  **关键选择**：不要点 OpenAI-compatible，而是往下拉，找到并点击 **【LocalAI】** 供应商。
3.  在弹出的对话框中，严格按照以下参数填写：
    *   **模型名称 (Model Name)**：`qwen3.6-35b-a3b-distilled-nothink`
        *(必须填大模型路由器 ini 中配置的真实 Section 标识符)*
    *   **模型类型 (Model Type)**：选择 `对话 (LLM)`
    *   **服务器 URL (Server URL)**：**`http://host.docker.internal:8080`**
        *(⚠️ 避坑神操作 1：使用 `host.docker.internal` 域名代替 `127.0.0.1` 才能跨越 Docker 访问宿主机)*
        *(⚠️ 避坑神操作 2：连路由器的正门 `8080`，不要连子进程的临时端口 `57987`)*
        *(⚠️ 避坑神操作 3：LocalAI 插件会自动拼接 `/v1`，因此这里**千万不要**加 `/v1` 尾巴，直接写到 `:8080` 截止)*
4.  点击 **添加 (Save)**，模型即可在 1 秒内通过验证并成功绑定！

## 🎯 任务三：在 Dify 工作室搭建“Obsidian 智能知识问答库”

现在，我们将“大脑”、“外挂硬盘”与 Dify 串联起来。

1.  **新建工作流**：在 Dify 主页进入“工作室” ➡️ 点击 **创建空白应用** ➡️ 选择 **Chatflow (工作流编排的聊天助手)** ➡️ 起名比如 `我的 Obsidian 智能大脑`。
2.  **配置 HTTP 请求节点 (外挂检索)**：
    *   在画布上，将鼠标放在“开始”节点右侧的加号上拉出连线，选择创建 **HTTP 请求 (HTTP Request)**。
    *   **方法 (Method)** 选 `POST`。
    *   **URL** 填入您的本地 RAG 请求地址（因为 Dify 在 Docker 内，所以同样必须使用 docker 域名）：
        `http://host.docker.internal:8000/retrieve`
    *   **Body** 选择 `JSON`，内容填：
        ```json
        {
          "query": "{{sys.query}}",
          "top_k": 3
        }
        ```
        *(说明：`{{sys.query}}` 代表聊天窗口里用户输入的问句。)*
3.  **连接大模型 (LLM) 节点**：
    *   将 HTTP 节点的输出拉线连接到 **LLM (大模型) 节点**。
    *   选择您在“任务二”中配置成功的 LocalAI 大模型 `qwen3.6-35b-a3b-distilled-nothink`。
    *   在 **SYSTEM (系统提示词)** 输入框内填入如下编排 Prompt：
        ```text
        你是一个专门根据本地 Obsidian 工作日志回答问题的智能助手。
        请结合以下由 RAG 系统检索出的参考知识片段，为用户提供极其精准的回复。如果找不到答案，请诚实说不知道，不要胡编乱造。
        
        【参考知识片段】：
        {{http_request.body.context}}
        ```
        *(说明：`{{http_request.body.context}}` 变量就是 RAG 接口以完美格式召回回来的 Markdown 文本段落。)*
4.  **连接直接回复节点**：将 LLM 节点的输出连线至“直接回复 (Answer)”节点。
5.  **发布运行**：点击右上角 **发布** ➡️ **在预览中运行**。

## 🏆 最终运行体验

在聊天框中提问：“我4月9号做过什么重构？”
1.  Dify 捕捉到问题，通过 **HTTP 请求** 发送给跑在 CPU 上的 **本地 RAG API**。
2.  RAG 服务去 Chroma 数据库检索到 `2026-04-09_Llama智能启动器重构.md` 中的精细父块（含完整无乱码的 YAML 标题和 Obsidian callout 块）。
3.  RAG 将文本段塞给 **LocalAI (Qwen3.6-35B)**。
4.  Qwen 大模型流式输出完美整理后的重构内容。

*至此，本地化大模型 ➕ 精准 Markdown 知识召回的闭环生态构建完毕！*
